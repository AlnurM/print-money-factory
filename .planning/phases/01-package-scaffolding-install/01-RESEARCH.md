## Summary

Phase 1 builds the entire installable foundation: an npm package that copies Claude Code slash commands to `~/.claude/commands/brrr/`, installs workflows/templates/references to `~/.pmf/`, creates a Python venv at `~/.pmf/venv/` with all backtest dependencies, and ships a fixed metrics module with unit tests and reference backtest engine patterns.

The npm side is a thin distribution layer -- a file copier with Python venv management. The GSD (get-shit-done) package provides the exact architectural pattern to mirror: `bin/install.js` as the npx entry point, commands/ as thin markdown files, workflows/ for full behavioral prompts, templates/ for scaffolding, references/ for fixed code and knowledge. The Python side includes the metrics module (the trust anchor for all downstream calculations) and reference patterns for data loading and backtest engine structure.

**Primary recommendation:** Keep the npm package simple (plain Node.js `fs` operations, no build step needed). Focus implementation effort on the metrics module (known-answer unit tests for every metric) and the anti-lookahead backtest engine pattern. These are the two highest-recovery-cost items -- if either is wrong, all downstream phases produce invalid results.

### Pattern 1: Thin Command / Fat Workflow (from GSD)

**What:** Each `/brrr:*` command is a thin markdown file (10-20 lines) with YAML frontmatter that delegates to a workflow file via `@` reference. The command defines the interface (name, description, allowed-tools, argument-hint). The workflow defines the behavior.

**When to use:** Every command. No exceptions.

**Example:**
```markdown
---
name: brrr:execute
description: Run AI-driven backtest optimization loop
argument-hint: "[--iterations N] [--fast] [--resume]"
allowed-tools:
  - Read
  - Write
  - Bash
  - Glob
---
<objective>
Run the backtest optimization loop for the current phase.
</objective>

<execution_context>
@~/.pmf/workflows/execute.md
@~/.pmf/references/backtest-engine.md
@~/.pmf/references/metrics.py
@~/.pmf/references/data_sources.py
</execution_context>

<context>
Flags: $ARGUMENTS
</context>

<process>
Execute the backtest workflow from @~/.pmf/workflows/execute.md
</process>
```

### Pattern 2: npm bin Entry Point for npx

**What:** The `package.json` `bin` field maps the package name to an executable script. When a user runs `npx print-money-factory install`, npm downloads the package and executes `bin/install.mjs` with `install` as `process.argv[2]`.

**When to use:** The install entry point.

**Example package.json:**
```json
{
  "name": "print-money-factory",
  "version": "1.0.0",
  "description": "Trading strategy development pipeline for Claude Code",
  "bin": {
    "print-money-factory": "./bin/install.mjs"
  },
  "files": [
    "bin/",
    "commands/",
    "workflows/",
    "templates/",
    "references/",
    "requirements.txt"
  ],
  "type": "module",
  "engines": {
    "node": ">=18"
  },
  "keywords": ["claude-code", "trading", "backtest", "pinescript"],
  "license": "MIT"

```

**Key details:**
- Use `.mjs` extension for the bin script to ensure ESM parsing regardless of consumer environment
- The `"type": "module"` field enables ESM imports in project files
- The `"files"` array explicitly lists what gets published (excludes dev files, tests, etc.)
- No build step needed -- ship plain JavaScript and markdown files

### Pattern 3: Idempotent Install Script

**What:** The install script can be run any number of times safely. First run creates everything. Subsequent runs update files and upgrade pip packages without destroying user additions.

**When to use:** Every install and update invocation.

**Logic flow:**
```
1. Parse command: expects "install" as first argument
2. Resolve paths: homedir + /.claude/commands/brrr/, homedir + /.pmf/
3. Create directories (mkdirSync recursive -- safe if exists)
4. Copy commands/*.md -> ~/.claude/commands/brrr/ (overwrite)
5. Copy workflows/, templates/, references/ -> ~/.pmf/ (overwrite)
6. Write ~/.pmf/.version with current version and date
7. Check Python: execSync("python3 --version") -> parse semver -> require >=3.10
8. If ~/.pmf/venv/ exists:
   - pip install --upgrade -r requirements.txt
9. Else:
   - python3 -m venv ~/.pmf/venv/
   - ~/.pmf/venv/bin/pip install -r requirements.txt
10. Verify: ~/.pmf/venv/bin/python -c "import pandas; import numpy; print('OK')"
11. Print success summary with file counts
```

**When to use:** Every generated backtest. This is the core pattern Claude must follow.

**Skeleton structure:**

### Anti-Patterns to Avoid

- **Logic in commands instead of workflows:** Commands must be thin (10-20 lines). All behavioral logic lives in workflows/ referenced via `@`. GSD enforces this strictly.
- **Postinstall hooks:** Do NOT use npm `postinstall` scripts. Many package managers block them. Use explicit `npx print-money-factory install` instead.
- **System Python:** Never use system Python or assume packages are globally installed. Always use `~/.pmf/venv/bin/python`.
- **Regenerating metrics each run:** The metrics module is FIXED code, not Claude-generated. Claude imports from it.
- **Building a backtest framework:** Do NOT build a generic backtesting library. Ship patterns and a skeleton. Claude writes custom code per strategy.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Financial metric calculations | Inline Sharpe/Sortino/MaxDD in each backtest | Fixed `metrics.py` module imported by all backtests | Annualization, downside deviation, drawdown have subtle edge cases (empty trade lists, single-bar trades, all-winning trades). Get it right once, test it, never rewrite. |
| Data source loading | Custom HTTP/exchange code per backtest | Fixed `data_sources.py` with `load_ccxt()`, `load_yfinance()`, `load_csv()` | Rate limiting, error handling, data validation, NaN handling -- each data source has its own gotchas. |
| Python version detection | Regex on `python --version` output | `execSync('python3 -c "import sys; print(sys.version_info[:2])"')` | Parsing version strings is fragile. Let Python report its own version tuple. |
| File copy with directory creation | Manual fs.mkdirSync + fs.copyFileSync chains | Recursive directory copy helper function | Cross-platform path handling, nested directory creation, error reporting. Write once in the install script. |
| PineScript generation | Claude writing PineScript from scratch each time | Template + examples per strategy type | PineScript v5 has strict syntax rules. Templates ensure valid structure. Examples show correct patterns by strategy archetype. |

**Key insight:** The metrics module and data source adapters are "trust anchors" -- if they are wrong, every downstream result is wrong. Hand-rolling these per run introduces inconsistency and bugs. Fixed, tested code eliminates this entire class of errors.

## Common Pitfalls

### Pitfall 1: ensurepip Missing on Debian/Ubuntu

**What goes wrong:** `python3 -m venv ~/.pmf/venv/` fails with "ensurepip is not available" on Debian-based systems where `python3-venv` is not installed by default.
**Why it happens:** Debian splits the Python standard library into multiple packages. The `venv` module requires `python3-venv` (or `python3.X-venv`) to be installed separately.
**How to avoid:** Catch the venv creation error and display: "Install python3-venv: sudo apt install python3-venv (or python3.X-venv for your Python version)"
**Warning signs:** Install fails on Ubuntu/Debian but works on macOS.

### Pitfall 2: Python 3 Not Available as `python3`

**What goes wrong:** Install script calls `python3` but on some systems (especially Windows WSL or older Linux), only `python` exists and points to Python 3.
**Why it happens:** Python binary naming is inconsistent across platforms.
**How to avoid:** Try `python3` first, fall back to `python`, verify version in both cases. On Windows, also try `py -3`.
**Warning signs:** "command not found: python3" on systems where Python 3 is installed.

### Pitfall 3: pip Install Fails Due to Compilation

**What goes wrong:** `pip install` fails when trying to compile C extensions for numpy/scipy/pandas on systems without build tools.
**Why it happens:** These packages ship binary wheels for common platforms, but edge cases (older OS, ARM Linux, musl-based distros) may require compilation.
**How to avoid:** Use `pip install --only-binary :all:` as first attempt. Fall back to normal pip install if that fails (some packages may not have wheels for the platform). Display clear error about installing build tools if both fail.
**Warning signs:** Errors mentioning `gcc`, `cl.exe`, or `compilation failed`.

### Pitfall 4: Sharpe Ratio Annualization Errors

**What goes wrong:** Sharpe ratio computed from daily returns is not annualized, or annualized with the wrong factor (252 for stocks vs 365 for crypto vs actual trading days).
**Why it happens:** The annualization factor depends on the data frequency. Using 252 for crypto data (which trades 365 days) or 365 for stock data (which has ~252 trading days) gives incorrect results.
**How to avoid:** The metrics module must accept a `trading_days_per_year` parameter (default 252). Callers specify based on asset class. Document this clearly.
**Warning signs:** Sharpe ratio seems too high or too low compared to published benchmarks for similar strategies.

### Pitfall 5: Lookahead Bias in Backtest Engine

**What goes wrong:** The backtest engine skeleton allows signal computation to see future data, or executes trades at the signal bar's price instead of the next bar's open.
**Why it happens:** It is the most natural way to write backtest code -- compute signal from current bar, execute at current bar's close. But this is cheating because you could not act on the close price until after the bar closes.
**How to avoid:** The skeleton enforces: signals use `df.iloc[:i+1]` (history up to current bar), execution uses `df.iloc[i+1]['open']` (next bar's open). The pattern guide explains why.
**Warning signs:** Sharpe > 3.0, win rate > 70% on trend-following, no drawdowns.

### Pitfall 6: Overwrite of User-Modified Files

**What goes wrong:** Running install/update overwrites user's custom modifications to template files or workflow files.
**Why it happens:** Idempotent install means overwriting files, but some files might have been intentionally modified by the user.
**How to avoid:** For Phase 1, overwrite is the correct behavior -- templates and references should match the package version. If user customization becomes a concern, add a `~/.pmf/.local/` directory for user overrides in a future version. Document that `~/.pmf/` is managed by the installer.
**Warning signs:** Users complain about losing changes after update.

## Code Examples

### Install Script Structure (bin/install.mjs)

```javascript
#!/usr/bin/env node
// Source: GSD pattern (bin/install.js) + project decisions

import { fileURLToPath } from 'url';
import { dirname, join, resolve } from 'path';
import { homedir } from 'os';
import { existsSync, mkdirSync, copyFileSync, readdirSync, statSync, readFileSync, writeFileSync } from 'fs';
import { execSync } from 'child_process';

const __filename = fileURLToPath(import.meta.url);
const __dirname = dirname(__filename);
const PKG_ROOT = resolve(__dirname, '..');

const HOME = homedir();
const COMMANDS_DIR = join(HOME, '.claude', 'commands', 'brrr');
const PMF_DIR = join(HOME, '.pmf');
const VENV_DIR = join(PMF_DIR, 'venv');
const PYTHON_BIN = join(VENV_DIR, 'bin', 'python');

function main() {
  const command = process.argv[2];
  if (command !== 'install') {
    console.error('Usage: npx print-money-factory install');
    process.exit(1);
  }

  console.log('Print Money Factory - Installing...\n');

  // Step 1: Copy commands
  copyDir(join(PKG_ROOT, 'commands'), COMMANDS_DIR);
  console.log(`  Commands -> ${COMMANDS_DIR}`);

  // Step 2: Copy workflows, templates, references
  for (const dir of ['workflows', 'templates', 'references']) {
    copyDir(join(PKG_ROOT, dir), join(PMF_DIR, dir));
    console.log(`  ${dir}/ -> ${PMF_DIR}/${dir}/`);
  }

  // Step 3: Write version file
  const pkg = JSON.parse(readFileSync(join(PKG_ROOT, 'package.json'), 'utf8'));
  writeFileSync(join(PMF_DIR, '.version'), JSON.stringify({
    version: pkg.version,
    installed: new Date().toISOString(),
    repo: pkg.repository?.url || 'https://github.com/user/print-money-factory'
  }, null, 2));

  // Step 4: Check Python
  const pythonPath = detectPython();

  // Step 5: Create or update venv
  if (existsSync(VENV_DIR)) {
    console.log('\n  Updating existing venv...');
    execSync(`"${PYTHON_BIN}" -m pip install --upgrade -r "${join(PKG_ROOT, 'requirements.txt')}"`,
      { stdio: 'inherit' });
  } else {
    console.log('\n  Creating Python venv...');
    execSync(`"${pythonPath}" -m venv "${VENV_DIR}"`, { stdio: 'inherit' });
    execSync(`"${PYTHON_BIN}" -m pip install -r "${join(PKG_ROOT, 'requirements.txt')}"`,
      { stdio: 'inherit' });
  }

  // Step 6: Verify
  execSync(`"${PYTHON_BIN}" -c "import pandas; import numpy; print('Dependencies OK')"`,
    { stdio: 'inherit' });

  console.log('\nInstall complete! Use /brrr:new-milestone to start.');
}
```

### Metrics Module Structure (references/metrics.py)

```python
# Source: Codearmo (Sharpe/Sortino/Calmar formulas), project decisions D-10 through D-14
"""
Fixed financial metrics module -- NOT regenerated by Claude.
Unit-tested with known-answer test cases.

Usage:
    from metrics import compute_all_metrics
    results = compute_all_metrics(trades=trade_list, equity_curve=equity)
    results = compute_all_metrics(daily_returns=returns_series)
"""
import numpy as np
import pandas as pd
from typing import Optional


def sharpe_ratio(returns: np.ndarray, risk_free_rate: float = 0.0,
                 trading_days: int = 252) -> float:
    """Annualized Sharpe ratio. Returns NaN if std is zero."""
    excess = returns - risk_free_rate / trading_days
    if len(excess) < 2 or np.std(excess, ddof=1) == 0:
        return float('nan')
    return float(np.mean(excess) / np.std(excess, ddof=1) * np.sqrt(trading_days))


def sortino_ratio(returns: np.ndarray, risk_free_rate: float = 0.0,
                  trading_days: int = 252) -> float:
    """Annualized Sortino ratio. Uses downside deviation."""
    excess = returns - risk_free_rate / trading_days
    downside = excess[excess < 0]
    if len(downside) < 2:
        return float('nan')
    downside_std = np.std(downside, ddof=1)
    if downside_std == 0:
        return float('nan')
    return float(np.mean(excess) / downside_std * np.sqrt(trading_days))


def max_drawdown(equity_curve: np.ndarray) -> float:
    """Maximum drawdown as a negative fraction (e.g., -0.15 = 15% drawdown)."""
    if len(equity_curve) < 2:
        return 0.0
    peak = np.maximum.accumulate(equity_curve)
    drawdown = (equity_curve - peak) / peak
    return float(np.min(drawdown))


def calmar_ratio(returns: np.ndarray, equity_curve: np.ndarray,
                 trading_days: int = 252) -> float:
    """Annualized return / abs(max drawdown)."""
    ann_return = float(np.mean(returns) * trading_days)
    mdd = max_drawdown(equity_curve)
    if mdd == 0:
        return float('nan')
    return float(ann_return / abs(mdd))


def win_rate(trades: list[dict]) -> float:
    """Fraction of profitable trades."""
    if not trades:
        return 0.0
    winners = sum(1 for t in trades if t['pnl'] > 0)
    return winners / len(trades)


def profit_factor(trades: list[dict]) -> float:
    """Gross profit / gross loss. Returns inf if no losing trades."""
    gross_profit = sum(t['pnl'] for t in trades if t['pnl'] > 0)
    gross_loss = abs(sum(t['pnl'] for t in trades if t['pnl'] < 0))
    if gross_loss == 0:
        return float('inf') if gross_profit > 0 else 0.0
    return gross_profit / gross_loss


def expectancy(trades: list[dict]) -> float:
    """Average P&L per trade."""
    if not trades:
        return 0.0
    return sum(t['pnl'] for t in trades) / len(trades)
```

```python
Known-answer tests for metrics module.
Run: ~/.pmf/venv/bin/python -m pytest ~/.pmf/references/test_metrics.py -v

class TestSharpeRatio:
    def test_known_answer_positive(self):
        # 252 days of 0.1% daily return, 0% risk-free
        returns = np.full(252, 0.001)
        # Expected: mean=0.001, std=0 -> NaN (no volatility)
        assert np.isnan(sharpe_ratio(returns))

    def test_known_answer_with_variance(self):
        # Alternating +1% and -0.5% daily returns
        returns = np.array([0.01, -0.005] * 126)
        # mean = 0.0025, std = ~0.0106, annualized ~ 0.0025/0.0106 * sqrt(252) ~ 3.74
        result = sharpe_ratio(returns)
        assert 3.5 < result < 4.0  # Approximate range

    def test_empty_returns(self):
        assert np.isnan(sharpe_ratio(np.array([])))


class TestMaxDrawdown:
    def test_known_answer(self):
        # Equity: 100 -> 120 -> 90 -> 110

    def test_no_drawdown(self):
```

## Open Questions

1. **Windows support for venv path**
   - What we know: `~/.pmf/venv/bin/python` is the Unix path. On Windows, it would be `~/.pmf/venv/Scripts/python.exe`.
   - What's unclear: Whether Windows/WSL users are a significant target audience.
   - Recommendation: Use `path.join(VENV_DIR, process.platform === 'win32' ? 'Scripts' : 'bin', process.platform === 'win32' ? 'python.exe' : 'python')` in install script. Workflow files can hardcode the Unix path since Claude Code primarily runs on macOS/Linux.

2. **GitHub version checking for /brrr:update**
   - What we know: D-04 says re-running install IS the update mechanism.
   - What's unclear: Whether there should be a version comparison step before re-running, or just always re-run.
   - Recommendation: For Phase 1, the update command simply runs `npx print-money-factory@latest install`. No separate version check API call needed. The `@latest` tag ensures the newest version is fetched.

3. **Test runner for metrics module in user environment**
   - What we know: pytest is not in requirements.txt (it is a dev dependency).
   - What's unclear: Whether users need to run tests themselves, or tests are only for development.
   - Recommendation: Include pytest in requirements.txt. The tests serve as both validation AND documentation. Users should be able to run `~/.pmf/venv/bin/python -m pytest ~/.pmf/references/test_metrics.py` to verify their installation.
