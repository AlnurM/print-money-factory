# Phase 1: Package Scaffolding & Install - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

<domain>
## Phase Boundary

npm package that installs via `npx print-money-factory install`. Copies slash commands to `~/.claude/commands/brrr/`, installs workflows/templates/references to `~/.pmf/`, creates a Python venv at `~/.pmf/venv/` with all backtest dependencies. Includes a fixed metrics module with unit tests and reference backtest engine patterns. Package structure mirrors GSD: commands/, workflows/, templates/, references/.

</domain>

<decisions>
## Implementation Decisions

### Install location
- **D-01:** Commands (thin markdown files) go to `~/.claude/commands/brrr/`
- **D-02:** Everything else (workflows, templates, references, bin) goes to `~/.pmf/`
- **D-03:** Commands @-reference workflow files from `~/.pmf/workflows/`
- **D-04:** `/brrr:update` works by running `npx print-money-factory install` again — idempotent install IS the update mechanism
- **D-05:** Install does NOT create `.pmf/` in the user's project directory — that's deferred to `/brrr:new-milestone` on first use

### Venv management
- **D-06:** Python venv lives at `~/.pmf/venv/` — inside the `~/.pmf/` directory, one place to manage
- **D-07:** On reinstall/update, existing venv gets `pip install --upgrade` — fast, preserves any user additions
- **D-08:** All Python execution uses full path `~/.pmf/venv/bin/python` — no activation needed, transparent to user
- **D-09:** Install detects Python 3.10+ before creating venv, fails with clear actionable error if missing

### Metrics module
- **D-10:** Fixed Python module at `~/.pmf/references/metrics.py` — NOT regenerated by Claude, unit-tested
- **D-11:** Accepts both trade log (list of trades with entry/exit time, price, size, side) AND daily returns series
- **D-12:** All core metrics in the fixed module: Sharpe, Sortino, Calmar, Max DD, Win Rate, Profit Factor, expectancy, trade count, net P&L
- **D-13:** IS/OOS split logic and overfitting detection are NOT in the metrics module — they live in the backtest workflow (Phase 4 concern)
- **D-14:** Module has comprehensive unit tests with known-answer test cases for each metric

### Reference patterns
- **D-15:** Backtest engine has BOTH: a pattern guide (markdown with anti-lookahead rules, next-bar execution requirement) AND a structural skeleton (backtest_engine.py with event loop, position management, metrics integration). Claude reads both — fills in strategy logic only.
- **D-16:** Data source adapters are ready-to-use Python code at `~/.pmf/references/data_sources.py` with tested functions: `load_ccxt()`, `load_yfinance()`, `load_csv()`. Claude calls these, doesn't rewrite them.
- **D-17:** PineScript has BOTH: a structural template (version header, input params, strategy() call structure) AND multiple complete examples by strategy type (trend-following, mean-reversion, breakout). Claude uses template as skeleton and examples as reference.

### Claude's Discretion
- Exact npm package.json structure and scripts
- install.js implementation details (error handling, progress display)
- Unit test framework choice for Python metrics
- Internal file organization within `~/.pmf/`
- requirements.txt exact version pins

</decisions>

<specifics>
## Specific Ideas

- "Mirrors GSD architecture" — the package structure should look and feel like GSD's commands/, workflows/, templates/, references/ pattern
- Data source adapters should be ready-to-use tested code, not patterns Claude rewrites each time
- The metrics module is a trust anchor — if metrics are wrong, everything downstream is wrong. Unit tests with known-answer cases are non-negotiable.
- The anti-lookahead pattern guide is critical — research identified this as the #1 risk for Claude-generated backtest code
